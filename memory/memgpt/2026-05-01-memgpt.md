# 把 LLM 的上下文当 RAM 来管——MemGPT 和它背后的操作系统哲学
## ——为什么 128k token 的上下文还不够，以及 2023 年的这个答案到 2026 年仍然没有被超越

> **论文**：MemGPT: Towards LLMs as Operating Systems
> **作者**：Charles Packer, Sarah Wooders, Kevin Lin, Vivian Fang, Shishir G. Patil, Ion Stoica, Joseph E. Gonzalez（UC Berkeley Sky Computing Lab）
> **发表**：2023 年 10 月 | arXiv:2310.08560 | NeurIPS 2023 Workshop
> **链接**：[论文](https://arxiv.org/abs/2310.08560) | [代码](https://github.com/cpacker/MemGPT) | [项目主页](https://research.memgpt.ai)
> **关键词**：`Virtual Context Management` `Memory Hierarchy` `Function Calling` `Long-term Memory` `跨 session 持久化`
> **一句话**：借用操作系统的内存分页思想，让 LLM 自己管理"把什么放进上下文窗口"，在跨 session 对话记忆任务上把 GPT-4 准确率从 32.1% 拉到 92.5%。

---

## TL;DR

**一句话总结**：MemGPT 把 LLM 的上下文窗口类比为操作系统的物理内存（RAM），把外部数据库类比为磁盘，通过让 LLM 自己调用函数来管理"什么数据在内存里、什么在磁盘上"，从而在不扩大上下文的前提下实现理论上无限的对话记忆。

**三点拆解**：

- 🔑 **OS 比喻不是噱头，是真正的工程框架**：MemGPT 不只是"加个 RAG"，而是精确复现了 OS 的内存层次结构——包括内存压力警告（warning threshold=70%）、页面替换（FIFO 驱逐 + 递归摘要）、以及中断驱动的控制流。这套框架让 LLM 不是被动接收上下文，而是主动决策"现在需要记住什么"。
- 🔑 **DMR 实验的 60 点差距说明了什么**：在跨 session 深度记忆检索任务中，MemGPT+GPT-4 达到 92.5% 准确率，而带递归摘要的纯 GPT-4 只有 32.1%——这个差距不是模型能力的差异，而是"有没有完整历史可查"的差异。递归摘要会丢失细节，向量搜索不丢。
- 🔑 **Function Chaining 是让这套架构可用的隐藏机制**：单次检索往往不够用——找到"第一层"答案后需要再查"第二层"。MemGPT 通过 `request_heartbeat=true` 让 LLM 在函数调用后立即再次推理，实现多跳检索。没有这个机制，嵌套 KV 检索任务（需要 4 次串联查询）在实验中完全无法完成。

## §3 背景与动机：上下文窗口是一个会溢出的桶

### 你的 Agent 有多健忘？

给你的 Agent 助理布置一个跨周任务：周一告诉它你的偏好、项目背景、团队成员名字；周四去问它上次讨论的结论。大概率它不记得了——不是因为它不够聪明，而是因为周四的请求开了一个全新的上下文窗口，周一的对话物理上已经不见了。

这是 LLM 架构的根本约束，不是 bug，是特性。Transformer 的 self-attention 在推理时必须把整个上下文一次性放进来。你放进去的 token 越多，计算量按 O(n²) 增长——这不是工程优化可以绕过的，是数学。所以每个模型都有一个硬上限：2k、4k、8k、32k、128k tokens。超了就截断；截断就失忆。

### "128k 够了吧？"不够

你可能觉得现在的大模型已经有了足够大的上下文窗口。GPT-4 Turbo 128k、Claude 2 100k——这已经是几十万字的容量了，一个长期对话助理还需要什么？

2023 年，斯坦福+华盛顿大学的研究（"Lost in the Middle"，Liu et al., 2023）给出了一个扎心的发现：**LLM 不善于利用长上下文中间部分的信息**。把答案放在开头或结尾，模型能找到；把它塞在 32k 的中间，正确率掉到接近随机。这意味着即使你买得起 128k 的 token 费用，把 100 页文档全部喂进去也未必比喂 5 页更有用。

更实际的问题是：**客服 Agent 一旦部署，就需要记住每个用户的历史**。如果一个用户和 Agent 对话了三年，1000 次会话，不可能把所有历史每次都放进 128k 的窗口——光是历史就能撑爆 5 个 128k 窗口。

### 早期方案为什么不够用

**方案一：递归摘要（Recursive Summarization）**

最直觉的做法：上下文快满了，就让 LLM 把旧消息压缩成一段摘要，摘要放回去，旧消息删掉。OpenAI 的早期对话助理、LangChain 的 `ConversationSummaryMemory` 都是这个思路。

问题在于摘要是有损的。"上周你说你不喜欢在会议里被打断"这样的具体细节，在三轮压缩后变成"用户有一些偏好"，再压一次就消失了。MemGPT 的 DMR 实验里，基线模型正是依赖递归摘要，最终准确率只有 32.1%——因为被问到的具体细节早就被摘要掉了。

**方案二：直接 RAG**

用向量数据库存历史，检索时召回相关片段。这是正确方向，但工程层面有几个坑：检索质量依赖嵌入质量；多跳问题（需要先找 A 再用 A 的结果找 B）在单次检索里无法处理；检索的结果塞回上下文后，上下文满了怎么办？——一个没有"主动内存管理"的 RAG 系统，是在用被动工具拼一个主动问题。

**时代背景**：2023 年，OpenAI 发布了支持 function calling 的 GPT-4，这是 MemGPT 成立的前提。没有可靠的函数调用能力，LLM 就无法自主管理自己的内存。这个时间窗口解释了为什么 MemGPT 在 2023 年出现，而不是更早。

更深层的背景是：2023 年正是"LLM Agent"从概念走向工程实践的转折年。ReAct、Toolformer 等工作已经证明了 LLM 可以使用工具，但这些系统都是无状态的——每次调用工具、获取结果，任务完成后状态消失。MemGPT 是第一批明确把"持久化状态管理"作为 Agent 核心能力来设计的系统之一。它的 UC Berkeley 团队背景（Sky Computing Lab，Ion Stoica 是 Apache Spark 的共同创始人）也解释了为什么这篇论文的思维框架如此偏向系统设计——他们看到的不是"如何让 LLM 记住更多"，而是"如何在有限资源约束下设计一个内存层次结构"，这两个视角导出的解决方案有根本性的差异。

## §4 核心 Idea：你的 LLM 就是 CPU，上下文就是 RAM

💡 **核心类比**：1960 年代操作系统发明了"虚拟内存"，让程序以为自己有无限内存——实际上 OS 在后台把不用的页面换出到硬盘，需要时再换回来。MemGPT 做的事情完全一样，只是把"程序"换成了 LLM，把"物理内存"换成了上下文窗口，把"硬盘"换成了数据库，把"OS 内存管理器"换成了 LLM 自己。

```
        ┌──────────────────────────────────────────────────────┐
        │         LLM 有限上下文窗口（如 8k tokens）             │
        │                                                      │
Prompt  │  ┌──────────────┐  ┌─────────────────┐  ┌────────┐ │
Tokens  │  │ System Instr │  │ Working Context  │  │  FIFO  │ │
        │  │  (只读/静态)  │  │(读写·存用户长期信息)│  │ Queue  │ │
        │  └──────────────┘  └─────────────────┘  └────────┘ │
        │                                                      │
Compl.  │  ┌───────────────────────────────────────────────┐  │
Tokens  │  │           Output Buffer（函数调用输出）          │  │
        │  └───────────────────────────────────────────────┘  │
        └──────────────────────────────────────────────────────┘
               ↕ Function calls                  ↕ 消息写入/驱逐
        ┌──────────────────┐         ┌───────────────────────┐
        │ Archival Storage │         │    Recall Storage      │
        │（向量数据库/任意  │         │（完整对话历史数据库）   │
        │  长度文本知识库） │         │                       │
        └──────────────────┘         └───────────────────────┘
```

（参考原论文 Figure 2 & Figure 3 自绘）

上图的数据流是这样的：LLM 每次推理时，它的 prompt tokens（左侧三个色块）就是它"当前能看到的一切"。当 LLM 发现需要查更多历史，它可以通过 output buffer 调用函数，从 recall storage 里搜索；当它需要存储新知识，它可以写入 archival storage；当它意识到上下文快满了，它可以主动更新 working context，把最重要的信息保留下来。

这套架构里 LLM 同时扮演了两个角色：它是**执行业务逻辑的 CPU**（回答问题、理解对话），也是**自己的内存管理器**（决定什么进内存、什么换到磁盘）。这两个角色的合并是 MemGPT 最反直觉的设计决策，后面 §7 会详细分析这个决策的风险。

## §5 方法拆解：五个组件，每个都有它不得不这样设计的理由

### 5.1 Main Context 三段式：为什么不做成一个大文本框？

Main context 被分成三个功能不同的区域，而不是一个统一的大文本块。这个设计乍看多此一举，但每段有不同的更新语义：

**System Instructions（只读）** 从不改变。它包含 MemGPT 的控制流说明、函数列表的 schema、以及 LLM 角色定位。做成只读有两个作用：其一，防止 LLM 不小心"覆盖自己的操作手册"——这在早期的 self-editing 实验里是真实出现过的失败模式；其二，让这段内容总是出现在上下文的最前面，利用了"Lost in the Middle"效应——模型对上下文头部的注意力最高，系统指令放这里最可靠。

**Working Context（读写）** 是用户长期信息的"主内存"。论文没有规定它的格式，实验里用的是自然语言段落（"User is interested in machine learning research, prefers concise answers..."）。它只能通过函数调用修改——不是通过普通对话流修改。这个设计的理由是隔离：如果 LLM 可以在生成回复的同时顺便更新 working context，就很难保证更新的原子性和可溯源性。强制走函数调用相当于给写入加了一个"commit"，每次更新都有明确的操作记录。

**FIFO Queue（读写）** 存当前会话的滚动消息历史。"FIFO"意味着旧消息会被驱逐出队列——但不是彻底消失，而是先存入 recall storage，同时用递归摘要在队列头部保留一个"已驱逐消息摘要"。摘要存在的意义不是替代原始消息，而是在 LLM 判断"我需不需要去查历史"时提供一个低成本的线索——类似目录页，帮助 LLM 决定值不值得翻页查原文。

### 5.2 External Context：为什么需要两个数据库？

MemGPT 有两个外部存储：recall storage（消息历史数据库）和 archival storage（知识数据库）。为什么不合并成一个？

两个库的检索语义不同。Recall storage 按时间和语义搜索对话历史——"上次用户提到什么"；archival storage 存任意长度的文档和知识——"Wikipedia 里关于 X 的条目"。前者是结构化的对话流，后者是非结构化的知识库。把它们合并会导致两种检索模式互相干扰：在找对话历史时会召回不相关的文档知识，反之亦然。论文设计了两套独立的函数接口：`conversation_search` 查 recall storage，`archival_memory_search` 查 archival storage，在 prompt 层面就明确了"你在查什么"。

从工程角度讲，这两个库还有不同的存储需求：recall storage 需要支持按时间排序和精确 session 过滤；archival storage 需要高效的向量相似度搜索、支持分页、可存储 MB 级别的文档。把它们合并成一个库要么让向量 DB 兼容时间排序（不擅长），要么让关系型 DB 兼容向量搜索（需要扩展）——两个独立库是最干净的工程选择。

### 5.3 Queue Manager 与内存压力：两阶段驱逐的状态机

OS 的页面替换算法（LRU、FIFO、Clock）核心逻辑是：当物理内存不够用时，选一个"最不可能被马上访问的页面"换出到磁盘。MemGPT 的 Queue Manager 复现了这个逻辑，但有一个聪明的改动：**在内存溢出之前就提前告警**。

```
 队列长度
     │
100% ├──────────────────────────────────── flush threshold
     │                          ↑ 强制驱逐 50% 消息
     │                          │ + 生成递归摘要
     │                          │ + 写入 recall storage
 70% ├───────────────────────── warning threshold
     │              ↑ 插入系统消息："内存压力警告，
     │              │  请将重要信息转存 working context
     │              │  或 archival storage"
  0% └──────────────────────────────────────────→ 时间
      正常写入     收到警告（LLM 可主动整理）   强制驱逐
```

（参考原论文 §2.2 Queue Manager 设计自绘）

这个两阶段设计（先警告再强制）是整套架构里工程细节最精致的部分。如果只有强制驱逐，LLM 会在毫无准备的情况下突然"失忆"；有了提前警告，LLM 可以用最后几个推理步骤做"内存整理"，把最重要的信息搬到 working context 或 archival storage 保存。被驱逐的消息从 LLM 的视野里消失了，但不会永远消失，随时可以通过 `conversation_search` 找回——就像 OS 把换出的页面保存在 swap 分区，等到 page fault 时再换入。

### 5.4 Function Executor：为什么用接口契约替代 Prompt 约定？

Function Executor 负责解析 LLM 的输出（completion tokens），提取函数调用，执行函数，并把结果反馈回上下文。

MemGPT 的函数集合设计有一个值得注意的约束：**所有内存操作都被包装为工具调用，而不是通过 prompt engineering 指令控制**。为什么？因为 function calling 接口有确定性更高的格式——GPT-4 的 function calling 会产生 JSON 格式的函数参数，比让 LLM 用自然语言描述"我要更新内存"然后再解析可靠得多。这是一个工程选择：用接口契约替代 prompt 约定，把"写入内存"从软性指令变成硬性 API 调用。

核心函数集合：
- `core_memory_append(key, content)`：向 working context 的指定字段追加信息
- `core_memory_replace(key, old_content, new_content)`：替换 working context 中的信息
- `archival_memory_insert(content)`：向 archival storage 插入新条目
- `archival_memory_search(query, page)`：按语义搜索 archival storage，支持分页
- `conversation_search(query, page)`：按语义搜索 recall storage

分页参数（`page`）是个关键细节。一次检索召回的结果可能有数十条，不能全部塞进上下文（会造成新的溢出），所以检索结果是分页的，LLM 可以翻页继续查——这正是 function chaining 机制发挥作用的场景。

### 5.5 Function Chaining：解决多跳检索的关键机制

普通的 LLM agent 调用一次工具后会等待用户输入——"一问一答"。但很多实际任务需要连续多步工具调用：先搜索一个 UUID，得到的值再搜索一次，如此循环 4 次。MemGPT 通过一个特殊的函数参数 `request_heartbeat=true` 来实现连续调用：当 LLM 在函数调用里加上这个参数，函数执行完毕后会立即触发下一轮 LLM 推理，而不是等用户输入。如果不加这个参数（即 yield），MemGPT 将不运行 LLM 推理，直到下一个外部事件（用户消息或调度中断）到来。

类比到 OS：`request_heartbeat=true` 相当于系统调用返回后立即执行下一条指令（而不是把控制权交回用户程序等待输入）。在嵌套 KV 检索实验里，这个机制直接决定成败：4 层嵌套需要 4 次串联的 `archival_memory_search` 调用，缺少 function chaining 就无法完成。实验数据也印证了这点：MemGPT+GPT-4 在 4 层嵌套下仍然保持接近 100% 的准确率，而 GPT-4 Turbo（上下文更大）在 3 层时就开始掉到 0%。

## §6 实验结果 + 代码示例

### 核心实验数据

**Table 2：Deep Memory Retrieval（跨 session 一致性）**

| 模型 | 准确率↑ | ROUGE-L (R)↑ |
|------|--------|-------------|
| GPT-3.5 Turbo（baseline，递归摘要） | 38.7% | 0.394 |
| GPT-3.5 Turbo + MemGPT | 66.9% | 0.629 |
| GPT-4（baseline） | 32.1% | 0.296 |
| GPT-4 + MemGPT | 92.5% | 0.814 |
| GPT-4 Turbo（baseline） | 35.3% | 0.359 |
| **GPT-4 Turbo + MemGPT** | **93.4%** | **0.827** |

baseline 是带递归摘要的固定上下文模型；MemGPT 有完整历史但通过搜索访问。

**DMR 评测集是怎么构造的？** 实验基于 Multi-Session Chat（MSC）数据集，包含五轮多 session 聊天记录，每轮约十几条来回。论文额外构造了第六轮——只有一个问答对，问题专门考察前五轮聊天里出现过的具体细节（比如"你还记得我们上次约定在哪里吃饭吗"），而且这些细节故意设计成不能从角色的 persona 摘要里推断出来，必须真正"翻过"原始对话才能回答。评判标准用的是 LLM judge（让 GPT-4 判断回答是否正确）配合 ROUGE-L 召回率。用 ROUGE-L 而不是精确匹配，是因为 LLM 生成的回复往往比 ground truth 更冗长——召回率度量"生成结果是否涵盖了 ground truth"，比精确字符串匹配更公平。

一个反常的数字：GPT-4 baseline 比 GPT-3.5 准确率更低（32.1% vs 38.7%）。可能原因之一是 GPT-4 更"谨慎"——当摘要信息不足以确定时，GPT-4 倾向于回答"我不知道"而不是猜测，反而让精确匹配的 accuracy 指标更低。另一种同等可能的解释是：GPT-4 生成的递归摘要压缩率更高——摘要更简洁意味着更多具体细节在摘要阶段被丢弃，导致答案的源材料本身就比 GPT-3.5 的摘要更稀疏。两种解释目前均无法从论文数据中区分，这本身也说明 DMR 评测在控制变量上存在缺陷：baseline 的"摘要质量"没有被独立控制，使得 GPT-4 和 GPT-3.5 的差异难以归因。这是一个值得警惕的评测陷阱：**在记忆质量评估中，"谨慎"的模型在正确率指标上看起来比"乱猜"的模型更差**，但前者的实际用户体验往往更好。

**Nested KV Retrieval（多跳检索能力）**

GPT-3.5 在 1 层嵌套时准确率直接降到 0%；GPT-4 在 3 层时降到 0%；MemGPT+GPT-4 在 4 层仍保持高准确率。这个任务的特殊性在于它排除了"智力"因素——UUID 没有语义，LLM 无法依靠知识猜测答案，只能老老实实逐层查询。失败的原因不是"想不明白"，而是"没有工具完成多步查询"。MemGPT 的 function chaining 在这里是纯粹的工程优势，和模型智力无关。

最反常的结论：MemGPT+GPT-4 Turbo 在嵌套 KV 上反而比 MemGPT+GPT-4 差——GPT-4 Turbo 的大上下文让它倾向于"直接在窗口里找"而不是迭代搜索，导致多跳任务失败。**更大的上下文不是更好的内存管理——它只是推迟了问题，顺带破坏了 Agent 的检索习惯**。

### 代码示例：复现 MemGPT 核心记忆管理逻辑

```python
"""
复现 MemGPT 的核心记忆管理机制
来源：Packer et al. 2023 - MemGPT: Towards LLMs as Operating Systems
依赖：pip install numpy
运行：python memgpt_demo.py
"""

import json
import time
from collections import deque
from dataclasses import dataclass, field
from typing import Optional

import numpy as np


# ─────────────────────────────────────────────
# 数据结构
# ─────────────────────────────────────────────

@dataclass
class Message:
    role: str       # "user" / "assistant" / "system"
    content: str
    timestamp: float = field(default_factory=time.time)


@dataclass
class ArchivalRecord:
    content: str
    embedding: np.ndarray
    created_at: float = field(default_factory=time.time)


# ─────────────────────────────────────────────
# External Context（磁盘层）
# ─────────────────────────────────────────────

def mock_embed(text: str) -> np.ndarray:
    """确定性 mock 嵌入（实际使用 text-embedding-ada-002 等）"""
    np.random.seed(hash(text) % 2**32)
    return np.random.randn(64).astype(np.float32)


class RecallStorage:
    """完整对话历史数据库（类比磁盘 swap 区）"""

    def __init__(self):
        self.messages: list[Message] = []

    def append(self, msg: Message):
        self.messages.append(msg)

    def search(self, query: str, page: int = 0, page_size: int = 5) -> list[Message]:
        q_embed = mock_embed(query)
        scored = []
        for msg in self.messages:
            sim = float(np.dot(q_embed, mock_embed(msg.content)) /
                       (np.linalg.norm(q_embed) * np.linalg.norm(mock_embed(msg.content)) + 1e-8))
            scored.append((sim, msg))
        scored.sort(key=lambda x: x[0], reverse=True)
        start = page * page_size
        return [m for _, m in scored[start:start + page_size]]


class ArchivalStorage:
    """任意长度文本知识库（类比磁盘持久存储）"""

    def __init__(self):
        self.records: list[ArchivalRecord] = []

    def insert(self, content: str):
        self.records.append(ArchivalRecord(content, mock_embed(content)))

    def search(self, query: str, page: int = 0, page_size: int = 5) -> list[ArchivalRecord]:
        """向量相似度检索，支持分页（function chaining 的基础）"""
        q_embed = mock_embed(query)
        scored = []
        for rec in self.records:
            sim = float(np.dot(q_embed, rec.embedding) /
                       (np.linalg.norm(q_embed) * np.linalg.norm(rec.embedding) + 1e-8))
            scored.append((sim, rec))
        scored.sort(key=lambda x: x[0], reverse=True)
        start = page * page_size
        return [r for _, r in scored[start:start + page_size]]


# ─────────────────────────────────────────────
# Main Context（内存层）
# ─────────────────────────────────────────────

class MemGPTContext:
    """
    MemGPT 主上下文管理器。
    WARNING_THRESHOLD: 触发内存压力警告的队列长度（对应 70% context window）
    FLUSH_THRESHOLD:   强制驱逐的队列长度（对应 100% context window）
    """
    WARNING_THRESHOLD = 8
    FLUSH_THRESHOLD   = 10

    def __init__(self, system_instructions: str):
        self.system_instructions = system_instructions     # 只读
        self.working_context: dict[str, str] = {}         # 读写，用户长期信息
        self.fifo_queue: deque[Message] = deque()         # 滚动消息历史
        self.recursive_summary: str = ""                   # 被驱逐消息的摘要
        self.recall_storage = RecallStorage()
        self.archival_storage = ArchivalStorage()

    # ── 函数调用接口（LLM 主动调用）──────────────

    def core_memory_append(self, key: str, value: str) -> str:
        """向 working context 追加信息"""
        self.working_context[key] = self.working_context.get(key, "") + " " + value
        return f"[OK] working_context['{key}'] updated"

    def core_memory_replace(self, key: str, new_value: str) -> str:
        self.working_context[key] = new_value
        return f"[OK] working_context['{key}'] replaced"

    def archival_memory_insert(self, content: str) -> str:
        self.archival_storage.insert(content)
        return f"[OK] Inserted: {content[:40]}..."

    def archival_memory_search(self, query: str, page: int = 0) -> str:
        """支持分页——function chaining 的核心应用场景"""
        results = self.archival_storage.search(query, page=page)
        if not results:
            return f"[NO RESULTS] page={page}"
        lines = [f"  [{i+1}] {r.content[:80]}" for i, r in enumerate(results)]
        return f"[Archival p={page}]\n" + "\n".join(lines)

    def conversation_search(self, query: str, page: int = 0) -> str:
        results = self.recall_storage.search(query, page=page)
        if not results:
            return f"[NO RESULTS] page={page}"
        lines = [f"  [{m.role}] {m.content[:80]}" for m in results]
        return f"[Recall p={page}]\n" + "\n".join(lines)

    # ── Queue Manager：内存压力控制 ───────────────

    def add_message(self, msg: Message) -> Optional[str]:
        """
        添加消息并触发内存压力检查。
        返回：None（正常）或 warning/flush 提示字符串
        """
        self.fifo_queue.append(msg)
        self.recall_storage.append(msg)  # 永远持久化到 recall storage

        if len(self.fifo_queue) >= self.FLUSH_THRESHOLD:
            return self._flush_queue()
        elif len(self.fifo_queue) >= self.WARNING_THRESHOLD:
            return "[SYSTEM] 内存压力警告：上下文已用 70%+，请将重要信息存入 working_context 或 archival_storage"
        return None

    def _flush_queue(self) -> str:
        """
        强制驱逐：弹出队列前半部分，生成递归摘要。
        被驱逐的消息仍在 recall_storage 中，可通过 conversation_search 找回。
        """
        evict_count = len(self.fifo_queue) // 2
        evicted = [self.fifo_queue.popleft() for _ in range(evict_count)]
        evicted_text = " | ".join(f"{m.role}: {m.content[:25]}" for m in evicted)
        self.recursive_summary = f"[摘要] {evicted_text}"  # 实际场景由 LLM 生成
        return f"[SYSTEM] 队列已刷新，{evict_count} 条消息移入 recall storage"


# ─────────────────────────────────────────────
# 演示：跨 session 记忆检索
# ─────────────────────────────────────────────

if __name__ == "__main__":
    print("=== MemGPT 记忆管理演示 ===\n")

    ctx = MemGPTContext("你是一个记忆助手，始终用 working context 记录用户的重要信息。")

    # Session 1：用户自我介绍
    print("--- Session 1 ---")
    session1 = [
        ("user",      "嗨，我叫小明，是 ML 工程师，在研究 RAG 系统。"),
        ("assistant", "你好小明！RAG 系统很有意思，遇到什么具体问题吗？"),
        ("user",      "主要是检索精度不够高，尤其是多跳问题。"),
        ("assistant", "明白，多跳检索需要迭代查询，我已记下你的背景。"),
    ]
    for role, content in session1:
        warning = ctx.add_message(Message(role=role, content=content))
        if warning:
            print(f"  ⚠️  {warning}")

    # LLM 主动调用函数存储重要信息（function chaining 简化演示）
    print(ctx.core_memory_append("user_profile", "名字：小明，职业：ML工程师，关注：RAG多跳检索"))
    ctx.archival_memory_insert("小明在 Session 1 提到主要问题是多跳检索精度不足")

    # Session 2：新窗口，FIFO Queue 清空，但 working context 和 archival storage 保留
    print("\n--- Session 2（新窗口，测试跨 session 记忆）---")
    ctx.fifo_queue.clear()  # 模拟新会话开始

    ctx.add_message(Message(role="user", content="上次我们聊到什么问题了？"))

    # LLM 先看 working context（已在 main context 里，无需调用函数）
    print(f"Working Context: {json.dumps(ctx.working_context, ensure_ascii=False)}")

    # LLM 再查 archival storage（function calling）
    print(f"\n{ctx.archival_memory_search('RAG 多跳检索')}")

    # 如需更多细节，翻查 recall storage（function chaining 翻页）
    print(f"\n{ctx.conversation_search('小明的问题')}")
```

**运行输出（确定性，hash 种子固定）：**
```

--- Session 2（新窗口，测试跨 session 记忆）---

[Archival p=0]
  [1] 小明在 Session 1 提到主要问题是多跳检索精度不足

[Recall p=0]
  [user] 主要是检索精度不够高，尤其是多跳问题。
  [user] 嗨，我叫小明，是 ML 工程师，在研究 RAG 系统。
  [assistant] 明白，多跳检索需要迭代查询，我已记下你的背景。
```

## §7 批判性分析：OS 比喻的边界在哪里

### 7.1 论文自承的局限

**GPT-3.5 的 function calling 能力不足**。MemGPT+GPT-3.5 在 DMR 任务上只从 38.7% 提升到 66.9%，比 GPT-4 的 92.5% 低了整整 26 个百分点。根本原因是 GPT-3.5 的函数调用格式稳定性差——它更容易输出格式错误的 JSON，导致函数执行失败，记忆操作悄悄地没有发生。这不只是"更强的模型效果更好"的平凡结论，而是说明 **MemGPT 架构对底层模型的函数调用可靠性有强依赖——架构的上限被模型能力锁死了**。

**更大上下文反而破坏检索习惯**。GPT-4 Turbo 在嵌套 KV 任务上的表现比 GPT-4 差，论文把这归因于 GPT-4 Turbo "在大窗口里直接找而不是迭代搜索"。论文描述了这个现象，但没有给出机制解释。我认为这揭示了一个更根本的设计缺陷：**MemGPT 没有任何机制强制 LLM 在上下文充足时仍然维持检索习惯——这套架构的正确运作依赖 LLM 的"自我约束"，而这在行为上是不稳定的**。只要模型版本更新或上下文窗口扩大，这个隐式约束就会静默失效，且没有任何监控能提前发现。这不是一个可以用 prompt tuning 稳定修复的问题，而是架构层面的脆弱性。

### 7.2 我看到的额外问题

**Working Context 的大小是一个未被讨论的关键超参数。** 论文对 working context 的大小几乎没有定量分析。太小的话重要信息放不下，LLM 会过度依赖 archival search（慢且贵）；太大的话挤压 FIFO Queue 容量，导致更频繁的驱逐——驱逐越频繁，递归摘要质量越差，长期记忆越容易退化。

更棘手的是，**working context 的更新策略完全依赖 LLM 的判断**。LLM 需要自己决定"什么信息值得放进 working context"，但如果它已经"忘记"了某条信息，它就无从判断那条信息是否重要——典型的"你不知道你不知道什么"。在长期运行的 Agent 里，working context 有很高概率逐渐退化：充斥着过时信息（"用户喜欢简洁的答案"，但用户早就改变了这个偏好），而 Agent 无法主动清理，因为清理需要知道哪些信息过时了，而判断"过时"需要历史对比，但那些历史已经被驱逐进了 recall storage。

**OS 比喻在"页面替换策略"上断层了。** MemGPT 的队列驱逐策略是 FIFO——把最老的消息驱逐出去。这忽略了一个关键点：重要性和时间新旧无关。一条一年前关于用户过敏信息的消息比五分钟前关于天气的闲聊重要得多，但 FIFO 无差别地按时间驱逐。Generative Agents 用 importance score 解决了这个问题——MemGPT 选择更简单的 FIFO，代价是驱逐策略次优，一些重要的历史信息可能在 recall storage 里永远没有被"想起来"的机会，因为 LLM 不知道它们存在，也没有任何机制提醒它去查。

**"LLM 即内存管理器"的单点故障风险。** 当 LLM 出现幻觉时，可能往 working context 里写入错误的信息（记错用户名字、记错偏好），而这个错误会持续存在——没有外部机制来校验 working context 的正确性。OS 里有内存保护机制（段错误、权限检查），MemGPT 里没有对应的设计。由于 working context 始终在上下文里，一条早期写入的错误信息会持续影响 LLM 的所有后续判断，直到 LLM 自己察觉并覆盖它——而察觉错误需要对比，对比需要记忆，这又绕回了同一个问题。

### 7.3 给工程师的实操建议

**把 working context 做成结构化 schema，而不是自由文本。** 把 working context 设计成固定 schema 的 JSON（`{ "user_name": "", "preferences": [], "ongoing_tasks": [] }`），每个字段有明确的更新语义，让 LLM 的更新操作从"随意写入"变成"填表格"，方便后续做一致性检查和外部监控。

**在 FIFO 驱逐之前加一个 importance 评分步骤。** 当 Queue Manager 触发警告时，主动对即将被驱逐的消息打 importance 分数，把高分消息自动保存到 archival storage——相当于把 Generative Agents 的 importance scoring 嫁接进 MemGPT 的驱逐流程，两篇论文的优点互补。额外成本：每次驱逐前多一次轻量 LLM 调用，在高频场景里可以用小模型替代 GPT-4 来控制成本。

**监控 function calling 的成功率，把它当核心指标。** 在生产部署里，记录每次 function call 的解析成功/失败率并设置告警阈值。如果成功率下降（可能是模型更新导致格式变化），整个记忆系统会悄悄失效——用户会注意到 Agent "越来越健忘"，但很难定位到函数调用格式的根本问题。失败的函数调用应该被记录并重试，而不是静默忽略。

## §8 应用落地 + 相关工作 + 资源

### 落地场景

**长期私人助理 / 企业知识助手**：一个为高管服务的 AI 助理需要记住这个人的工作风格偏好、历史决策背景、团队人员信息，而这些信息跨越数月、数年的对话积累——远超任何模型的上下文窗口。MemGPT 的跨 session 持久化机制直接解决这个需求，且不需要修改底层模型，可以即插即用地包装任何支持 function calling 的 LLM API。**主要部署坑**：working context 的序列化和反序列化容易被忽视——如果你在每次会话结束时把 working context 存进数据库、下次会话开始时再加载回来，就需要保证 JSON 的格式和字段名在版本迭代中保持向后兼容。一次 schema 变更如果没有迁移脚本，会导致所有历史用户的记忆在下次会话时被静默清空。

**超长文档分析 / 法律合同审查**：单文档超过 100 万 token 时，MemGPT 的 archival storage + 迭代检索允许 LLM 通过多跳检索"跨章节"回答问题（"第 3 章的定义和第 7 章的条款有没有冲突"），而不需要把整个文档一次性放进上下文。实验数据显示，在文档数量增加时 MemGPT 的性能不降反稳，而固定上下文基线随截断比例增大而退化。**主要部署坑**：文档分块策略对最终效果的影响远比 MemGPT 架构本身大。如果把一份合同按固定 512 token 切块，关键条款可能被截断在两个 chunk 的边界上，向量检索永远召回不到完整的语义单元。建议用语义分块（按章节、条款编号、段落边界）而不是按 token 数截断。

**游戏 NPC 的角色记忆**：与 Generative Agents 不同，MemGPT 更适合"单个 NPC 的深度记忆"——记住玩家所有历史交互（帮助、债务、诺言）。Working context 天然适合存储"关系状态"，recall storage 保留所有历史细节随时可查。**主要部署坑**：游戏场景里 NPC 的记忆更新频率远比对话助理高——每次玩家交互都可能触发记忆写入，而写入操作需要 LLM 函数调用，这意味着每次交互至少一次额外的 API 调用。在实时游戏场景里，这会引入 200-500ms 的延迟，在玩家打断 NPC 说话时尤为明显。解决方案是把记忆写入异步化：交互时先用临时缓冲区记录，交互结束后批量写入，而不是实时写入。

### 2026 年：为什么副标题里的论断成立

标题说"2026 年仍然没有被超越"，这需要解释。

MemGPT 的核心贡献不是某个具体的基准分数，而是**一套把 OS 内存管理移植到 LLM 的工程框架**——main context / external context 的分层设计、函数调用驱动的自主内存管理、两阶段驱逐策略。这套框架在 2023 年提出后，后续出现的工作大多是在这个框架上的改进和工程化，而不是根本性的替代：Letta（MemGPT 的继任者）解决了生产部署问题；LangGraph 等框架提供了更灵活的 Agent 图调度；各种 memory 模块提供了更好的向量检索。但"让 LLM 自己决定什么进内存、什么在磁盘"这个核心 idea，以及分层存储 + function calling 的实现路径，至今没有出现一个公认的更好替代方案。截至 2026 年，Letta 框架仍然采用与 MemGPT 论文高度一致的架构，只是在工程层面做了打磨——这本身就是最好的佐证。

### 相关工作速览

| 工作 | 与 MemGPT 的核心区别 | 工程师的选择判据 |
|------|---------------------|----------------|
| **Generative Agents**（Park et al., 2023）| 三维检索（时近性+重要性+相关性）+ 反思机制，聚焦多 Agent 社会涌现，不处理跨 session 持久化 | 需要多 Agent 互动和社会涌现时选它；只需单 Agent 长期记忆时 MemGPT 更简洁 |
| **Reflexion**（Shinn et al., 2023）| 记忆存的是"上次失败的教训"用于任务纠错，不是积累经历的通用长期记忆 | 目标是提升特定任务的成功率（代码生成、问答）时用 Reflexion；目标是跨 session 记住用户时用 MemGPT |
| **FLARE**（Jiang et al., 2023）| 主动检索增强生成，在生成时决定"何时检索"，单次检索，无内存分层和 function chaining | FLARE 适合单次长文档问答；MemGPT 适合需要多跳或跨 session 的场景 |
| **LangChain ConversationMemory**（框架）| 提供对话历史存储原语（Summary、Buffer、VectorStore），但无 importance 评分、无 function chaining、无驱逐策略 | LangChain Memory 是"工具箱"，MemGPT 是"完整 OS"；前者更灵活，后者更开箱即用 |
| **Letta（MemGPT 后续）**（2024）| MemGPT 的工程继承者，加入 sleep-time compute（空闲时异步思考）和 portable memory（跨模型迁移），去掉了递归摘要 | 如果要在生产环境部署 MemGPT 风格的 Agent，直接用 Letta 而不是从论文代码重新实现 |

MemGPT 用一个工程师能立刻理解的类比（OS 虚拟内存）把一个难以描述的问题（LLM 如何管理无限对话历史）变成了一套可执行的架构。如果你今天要给自己的 Agent 加长期记忆，MemGPT 给你一个最小可用的框架；但在你真正部署之前，值得想清楚的问题是：**当 LLM 开始"管理自己的记忆"时，你怎么知道它管对了？** 2026 年最接近答案的方向，是把"内存管理"从 LLM 的自主行为变成一个有外部监控的系统行为——让 LLM 管内存，但人来审计内存的健康状态。Letta 的 sleep-time compute 是这个方向的第一步，但它解决的是"何时思考"，还没有解决"怎么知道记忆内容是对的"。
